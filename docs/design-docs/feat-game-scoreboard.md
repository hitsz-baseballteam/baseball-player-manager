# 比赛记分板 (Game Scoreboard) 实现计划

## Context

当前比赛数据录入需要逐球员手动填写 19 项统计字段（一场比赛 200+ 次数值输入）。用户需要一个**棒球记分卡式**的录入体验：从已有 Scenario（阵容方案）自动载入先发防守和打击棒次，然后按打席（plate appearance）点击结果按钮来记录比赛，系统自动推导出完整的 `PlayerGameStatLine`，最终生成 `Game` 对象存入 workspace，被数据中心和名人堂直接消费。

额外需求：训练赛（队内红白战）时，需要**左右双栏**同时记录 A 队和 B 队的数据。

---

## 核心理念

**三个阶段的比赛记录状态机：**

```
SETUP（赛前准备）→ RECORDING（赛中记录）→ REVIEW（赛后确认）→ 保存到 Workspace
```

- **SETUP**: 选日期/对手/比赛类型 → 选 Scenario → 自动填入先发防守 + 打击棒次 + 替补名单
- **RECORDING**: 逐打席点击结果按钮（一安/二安/本垒打/三振/…），系统自动推进棒次、计算跑者推进、累加统计数据
- **REVIEW**: 查看推导出的数据表，手动补充投手自责分/胜败救援，最终确认 → 生成 Game 对象 → 写入 workspace

## 核心数据流

### 打席结果 → 统计数据推导引擎

**打击结果按钮（14 种）**：

```
击球结果:   [一安] [二安] [三安] [本垒打]
保送/死球:  [四坏] [触身]
出局方式:   [三振] [滚地] [飞球] [平飞]
特殊情况:   [双杀] [牺飞] [牺触] [失误上垒] [野选]
```

**推导规则（核心算法）**：

| 结果 | PA | AB | H | HR/2B/3B | BB | SO | RBI | 出局数 | 跑者推进 |
|------|-----|-----|-----|-----------|-----|-----|------|--------|-----------|
| 一安 | +1 | +1 | +1 | - | - | - | 按得分 | 0 | 所有跑者+1垒 |
| 二安 | +1 | +1 | +1 | 2B+1 | - | - | 按得分 | 0 | 所有跑者+2垒 |
| 三安 | +1 | +1 | +1 | 3B+1 | - | - | 按得分 | 0 | 所有跑者+3垒 |
| 本垒打 | +1 | +1 | +1 | HR+1 | - | - | 跑者数+1 | 0 | 全部得分 |
| 四坏 | +1 | - | - | - | +1 | - | 按得分 | 0 | 受迫进垒 |
| 触身 | +1 | - | - | - | - | - | 按得分 | 0 | 受迫进垒 |
| 三振 | +1 | +1 | - | - | - | +1 | - | 1 | 不动 |
| 滚地 | +1 | +1 | - | - | - | - | 按得分 | 1 | 所有跑者+1 |
| 飞球 | +1 | +1 | - | - | - | - | 按得分 | 1 | 不动 |
| 平飞 | +1 | +1 | - | - | - | - | - | 1 | 不动 |
| 双杀 | +1 | +1 | - | - | - | - | 按得分 | 2 | 前2跑者出局 |
| 牺飞 | +1 | - | - | - | - | - | 三垒跑者得分 | 1 | 三垒得分 |
| 牺触 | +1 | - | - | - | - | - | 三垒跑者得分 | 1 | 所有跑者+1 |
| 失误上垒 | +1 | +1 | - | - | - | - | - | 0 | 所有跑者+1 |
| 野选 | +1 | +1 | - | - | - | - | 按得分 | 1 | 前位跑者出局 |

**跑者推进自动计算** (`computeRunnerAdvancement`)：
- 输入：打击结果 + 当前跑者位置 + 当前出局数
- 输出：新跑者位置 + 得分人数 + 打者打点数
- 这是一个纯函数，所有推导规则写死在代码中

**守备数据自动分配** (`distributeFieldingStats`)：
- 三振 → P 接杀 +1 PO, C +1 PO
- 滚地球 → 1B +1 PO, IF 轮转 +1 A
- 飞球 → OF 轮转 +1 PO
- 双杀 → 两个 IF 各 +1 A, 1B +1 PO
- 失误 → 随机野手 +1 E

### 投手数据

- 比赛开始时自动设定 P 位置球员为先发投手
- 「更换投手」按钮随时切换，记录换投时间点
- 局数 (IP) 根据每个投手在投期间的出局数自动计算
- 自责分 (ER)、投球数 (NP)、胜败救援 (W/L/SV) 在 REVIEW 阶段手动填入

### 比赛结束 → Game 对象生成

`finalizeGame()` 函数将 `ScoreboardGame`（运行态）转换为 `Game`（持久态）：
1. 遍历所有打席，聚合为每个球员的 `PlayerGameStatLine`
2. 按局汇总为 `InningRecord[]`（每局的安打/得分/上场打者）
3. 组合投手的 IP/ER/W/L/SV/NP
4. 输出完整的 Game 对象 → cloneWorkspace → push to games → save

## 双模式设计

### 标准模式（对外正式比赛）
- 单个 Scorecard 组件，全宽
- 只记录我方进攻数据
- 投球/守备数据也针对我方

### 双模式（队内训练赛）
- 两个 Scorecard 组件并排：`grid-template-columns: 1fr 1fr`
- A 队和 B 队各有独立的打击顺序、防守阵容、投手
- 两个队伍共用同一个球员池，但一个球员不能同时出现在两队
- 每局上半 A 队进攻（B 队记分卡灰掉），3 出局后自动切换到下半 B 队进攻
- 两个队伍的数据最终各自生成独立的 Game 对象

## 关键技术决策

1. **运行态与持久态分离**：`ScoreboardGame`（React state）只在比赛记录期间存在，finalize 时才转为 Game 写入 workspace。避免每次打席都写库。
2. **崩溃恢复**：每 5 个打席自动存 localStorage，下次打开页面提示「检测到未完成的比赛，是否恢复？」
3. **跑者推进简化**：规则覆盖 95% 真实棒球场景。极端情况（外野回传失误导致额外推进）通过「手动编辑跑者位置」处理。
4. **自责分手动**：ER 的判定需要还原无失误局面，MVP 阶段保持手动输入。
5. **盗垒为高级操作**：不在主按钮面板上，通过额外的「盗垒」按钮触发子操作。
