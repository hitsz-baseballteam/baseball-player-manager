# ADR-004: 选择乐观并发而非悲观锁

状态：已采纳

## 背景

共享 Workspace 仍然通过单个逻辑版本号协调并发写入。虽然当前底层已经从单表 `jsonb` 切换到归一化 `app_*` 表，但多个浏览器标签页或快速连续操作仍会竞争同一个 `app_workspace_meta.version`。需要一种并发控制策略。

## 决策

使用乐观并发控制：workspace meta 行持有 `version` 整数列，写操作先按 `slug + version` 锁定当前 meta 行，再仅在版本仍匹配时提交改动。冲突时返回 `409`，客户端重新加载最新数据并重新应用更改。

## 理由

- **低冲突概率**：项目是单人使用工具，多标签页同时编辑同一 workspace 的概率极低
- **版本语义清晰**：冲突边界集中在 `app_workspace_meta.version`，客户端和 API 都容易理解
- **实现仍然直接**：即便当前实现用事务 + `SELECT ... FOR UPDATE` 保护写窗口，真正的冲突判定仍然依赖版本匹配，而不是“谁先拿到长期锁谁赢”
- **用户可见冲突**：`409` 会触发 toast 提示、自动刷新或重放逻辑，用户知道发生了什么

## 备选方案

- **纯悲观锁**：适合高冲突场景，但会把“正确性”建立在锁持有时机上。当前项目仍然只在短写事务中使用 `FOR UPDATE` 缩小竞争窗口，而不是把它作为主要并发模型
- **last-write-wins**：无声覆盖，简单但不安全。用户可能丢失数据而不自知

## 后果

- 冲突时仍需要客户端处理（重读最新数据并决定是否重放更改）
- 无自动字段级合并：冲突后用户操作可能丢失，需要重新操作
- 当前客户端通过 `submitMutationWithRetry()` 做有限重放；并发策略仍然偏向“检测并重试”，而不是“静默合并”
