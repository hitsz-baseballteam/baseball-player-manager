# ADR-004: 选择乐观并发而非悲观锁

状态:已采纳

## 背景

Workspace 数据存储在单表单行中。多个浏览器标签页或快速连续操作可能导致写冲突。需要一种并发控制策略。

## 决策

使用乐观并发控制:每行有一个 `version` 整数列,写操作通过 `WHERE version = $N` 检查版本。冲突时返回 409,客户端重新加载最新数据并重新应用更改。

## 理由

- **低冲突概率**:项目是单人使用工具,多标签页同时编辑同一 workspace 的概率极低
- **无锁开销**:不需要 `SELECT ... FOR UPDATE` 持有行锁,不会阻塞读操作
- **实现简单**:`UPDATE ... WHERE slug = $1 AND version = $2 RETURNING version`,失败时返回 null
- **用户可见冲突**:409 错误触发 toast 通知 + 自动刷新,用户知道发生了什么

## 备选方案

- **悲观锁**(`SELECT ... FOR UPDATE`):适合高冲突场景,但需要事务管理。单人工具中冲突极少,悲观锁会不必要地阻塞读取
- **last-write-wins**:无声覆盖,简单但不安全。用户可能丢失数据而不自知

## 后果

- 冲突时仍需要客户端处理（重读最新数据并决定是否重放更改）
- 无自动合并：冲突后用户操作可能丢失，需要重新操作
- 主工作区编辑路径已有 `saveWithRetry()` 自动重试；其他编辑路径仍可能需要用户重新应用修改
