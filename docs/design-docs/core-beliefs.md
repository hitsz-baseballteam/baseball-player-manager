# 核心信念

本项目遵循以下 Agent-First 工程原则。

## 1. 代码仓库是唯一的真相来源

任何信息如果只存在于聊天记录或人脑中，在运行时是不可见的。所有设计决策、产品规格、计划都必须在仓库中有文档记录。

**实践**：每个 PR 必须更新对应文档；代码注释解释"为什么"而非"是什么"。

## 2. 类型是契约，不是装饰

TypeScript 类型定义了系统边界。不在各处散落防御性 null 检查，而是在边界处验证和解析数据形状。

**实践**：
- 领域核心类型优先收敛在 `workspace.ts`；UI handle、API snapshot 等边界类型可在各自模块内定义
- `sanitizeWorkspace()` 在每次数据进入时执行
- 不要用 `as any` / `@ts-ignore` 绕过类型系统

## 3. 每个变更自包含

实现 + 测试 + 文档必须一起提交。不让下一个人（或明天的自己）去补测试和文档。

**实践**：
- 新功能必须包含对应测试（`*.test.ts` 或 `*.test.tsx`）
- 新 API 必须更新对应文档
- 破坏性变更必须记录在 design-docs

## 4. 直接 > 巧妙

优先选择实现简单、意图清晰的方案，而非过度抽象的"通用"方案。项目是单人工具，不需要为未知的扩展场景预先设计。

**实践**：
- 单表足够时不要拆多表
- 一个 `workspace.ts` 文件足够时不要拆十几个文件
- 纯函数足够时不要引入状态管理库

## 5. 数据安全优先级最高

用户的数据（球员档案、阵容方案）不可替代。任何操作如果不能保证数据完整性，宁可不做。

**实践**：
- 所有写操作通过乐观并发控制
- 所有数据入口经过 sanitizer
- 导入操作提供预览再确认

## 6. 逐步迁移 > 大爆炸重写

旧系统优先逐步迁移，只有在新页面与共享逻辑已经覆盖原能力后，才删除旧运行时。目标不是长期共存，而是**有证据地完成退役**。

**实践**：
- 新功能优先用 React 实现
- 共享业务逻辑优先沉淀为纯函数模块，避免 UI 迁移时复制规则
- 当旧运行时已无真实调用方时，应及时删除，而不是长期保留“备用”代码
- 迁移与退役过程都记录在 design-docs / exec-plans / tech-debt-tracker

## 7. 测试是活文档

测试不仅验证正确性，也是系统行为的活文档。新加入的人可以通过读测试理解每个函数应该做什么。

**实践**：
- 测试文件名使用 `*.test.ts` / `*.test.tsx` 并尽量贴近对应源文件
- 测试描述优先中文，但允许保留已有英文用例；关键是行为表达清晰
- 覆盖率不必追求 100%，但核心路径必须有测试
