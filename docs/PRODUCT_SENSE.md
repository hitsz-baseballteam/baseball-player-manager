# 产品判断力

## 核心原则

### 这是一个单人教练工具，不是社交平台

所有功能决策应以"一个棒球教练管理一支球队"为场景。不需要：

- 多用户 / 权限系统
- 实时协作
- 通知 / 消息
- 社交分享

### 数据永远比功能重要

用户的核心资产是球员数据（姓名、能力、雷达图）和方案数据（守位、棒次）。任何功能如果可能破坏数据完整性，优先级必须低于数据安全。

### 简单 > 强大

优先提供少量但可靠的功能，而非大量但半成品。每个新增功能必须：

1. 完成度 100%（无 [WIP] 标记上线）
2. 有对应的数据校验（非法输入不会破坏状态）
3. 可撤销（删除 / 恢复 / 导入导出）

## 功能优先级判断

| 优先级 | 标准 | 示例 |
|---|---|---|
| P0 必做 | 数据丢失风险 / 核心流程断裂 | 数据校验、版本冲突修复、导入导出 |
| P1 该做 | 显著提升日常使用效率 | 自动排阵优化、批量球员编辑 |
| P2 可做 | 锦上添花但不影响核心任务 | 数据统计面板、导出 PDF |
| P3 不做 | 偏离核心场景 | 比赛实时记分、AI 阵容推荐 |

## UI 品质标准

### 好的 UI

- 反馈即时（点击有响应，保存有确认）
- 状态可见（当前在哪个方案、哪些球员已选）
- 操作可逆（删除有确认，导入不覆盖前可以预览）

### 避免的 UI

- 模态套模态（对话框里再弹对话框）
- 静默失败（操作后无任何反馈）
- 隐藏功能（需要长按 / 右键才能发现的操作）

## 球员档案的设计哲学

球员不只是数据行。档案编辑器（`PlayerProfileEditor`）是一张"球探卡片"：

- 雷达图用 SVG 手绘，而非图表库生成——体现工具的专业感
- 支持投手和野手两种模型切换——适应不同位置需求
- 信息密度高但不拥挤——分组折叠 + 颜色分区

## 导入导出的产品思考

- Workspace 导入是"搬家"——整库替换，适用于换设备
- Scenario 导入是"借东西"——增量追加，适用于分享阵容
- 始终让用户看到导入预览（球员列表 + 方案名称），再确认执行
- 导入后自动激活新方案（减少一步操作）

## 不做的事情

- AI 自动生成阵容：教练的判断力不可替代
- 移动端 App：当前 Web 响应式足够（解锁卡片已有 `min()` 适配）
- 实时记分 / 逐球事件采集：当前版本已有数据中心与比赛记录，但不做比赛现场记分系统
