# 20260604 前端重构蓝图：Baseball Player Manager

## 目标
为整个球员管理系统制定一套高辨识度、可分阶段落地的前端重构蓝图：既重做信息架构与页面组织，也定义明确的视觉方向、交互优先级与实施顺序，避免“首页还是旧壳、详情页是新壳”的割裂体验。

## 当前事实（基于仓库）
- `/` 仍以 `PlayerManagerClient` + `player-manager-dom.ts` 挂载 legacy manager 为主。
- `/players/[playerId]` 已经是较新的 React 页面，适合作为新设计语言的试验田。
- 当前 UI 是 React + legacy DOM 混合架构，不能假设一次性全量重写。
- 项目已有主题系统、帮助抽屉、引导浮层、档案页、解锁页与本地自托管字体能力。

## 设计总方向（推荐）
### 方向名
**Team Ops Editorial / 球队作战指挥台**

### 设计语气
- 不是普通 SaaS 后台
- 不是花哨电竞风
- 而是“教练战术板 + 体育年鉴 + 比赛日控制台”的混合体

### 关键词
- editorial（编排感强）
- athletic（有运动张力）
- strategic（像战术系统而非表单系统）
- tactile（卡片、标线、编号、状态标签有实体感）
- high-contrast（层级明确，能快速扫读）

### 让人记住的一点
**首页像比赛日前的球队作战台，而不是普通数据列表。**

## 信息架构（全站）
### 一级导航
1. **总览** `/`
2. **名册** `/roster`
3. **排阵** `/lineup`
4. **战术场景** `/scenarios`
5. **球员档案** `/players/[playerId]`
6. **数据中心** `/import-export`
7. **设置与帮助** `/settings`

### 页面职责
#### 1. 总览 `/`
系统中枢 / 比赛日总控台。
- 今日状态：可用人数、缺口位置、风险提醒
- 快捷动作：新增球员、导入、自动排阵、切换场景
- 阵容摘要：首发、守备、替补概览
- 关键提醒：伤病、人数不足、位置冲突
- 最近变更：最近编辑的球员 / 最近切换的方案

#### 2. 名册 `/roster`
全队球员目录，强调查找、筛选、批量操作。
- 列表 / 卡片双视图
- 位置、投打、可用性、年级筛选
- 批量编辑标签、状态、可用性
- 快速跳转球员档案

#### 3. 排阵 `/lineup`
真正的战术布置页面。
- 首发打线
- 守备位置图
- 替补与轮换
- 拖拽调整
- 自动排阵与冲突提示

#### 4. 战术场景 `/scenarios`
保存不同比赛条件下的方案。
- 对强投 / 对弱投 / 领先 / 落后 / 练习赛等模板
- 场景对比
- 一键套用到当前排阵

#### 5. 球员档案 `/players/[playerId]`
单球员深度信息页。
- 基础信息
- 位置能力与偏好
- 当前状态、备注、风险
- 与当前排阵的关系
- 历史修改摘要（如后续补）

#### 6. 数据中心 `/import-export`
低频但重要。
- 导入 JSON/CSV
- 导出当前工作区
- 备份/恢复
- 数据结构说明

#### 7. 设置 `/settings`
- 主题
- 共享口令相关说明
- 引导与帮助
- 视觉偏好 / 密度偏好（后续）

## 页面组织原则
### 原则 1：高频页面强中心化
高频任务集中在 3 个页面：
- 总览
- 名册
- 排阵

### 原则 2：详情页深挖，列表页快进快出
- 列表页负责筛选、选择、批量操作
- 详情页负责完整编辑和深度信息

### 原则 3：抽屉优先，整页保留给真正复杂任务
- 快速编辑：抽屉
- 深度编辑：独立页
- 低频复杂任务：独立页

### 原则 4：视觉层级服务于决策速度
优先让用户在 3 秒内知道：
- 今天能不能排出完整阵容
- 谁不可用
- 哪些位置缺人
- 当前方案是否安全

## 视觉系统计划（按 frontend-design 思路）
### 推荐视觉母题
**“战术板 + 年鉴”**

### 视觉要素
- 强标题区：编号、赛季、场景名、状态标签
- 局部网格打破：摘要卡、排阵板、球员标签错位排布
- 信息层：纸张感底色 + 运动场线条感 + 金属/油墨感强调色
- 轻动效：卡片进场、状态灯、切页滑动、hover 高亮联动

### 字体策略
- 当前仓库已自托管 `Inter` + `Noto Sans SC`，短期可延续
- 若要完全贯彻 frontend-design 方向，后续可在**标题字体**层面评估更有性格的本地自托管开源字体
- 正文仍需保持高可读性与中文兼容

### 配色策略
建议保留棒球/球场语义，但升级成更有“战术台”气质的系统：
- 主色：深墨绿 / 深松石 / 夜场蓝绿
- 强调色：旧金属金 / 红土橙 / 记号笔奶油白
- 风险色：锈红，不用默认 web 红
- 背景：避免纯白后台，采用带温度的浅纸色或夜场深底

## 实施策略（避免一次性重写）
- **不直接推翻 legacy manager**，而是先用 React 壳层统一视觉和导航
- 先把“入口感知最强”的区域替换掉：首页、导航、摘要、详情页
- 再逐步蚕食 legacy DOM 的内部功能区
- 每一阶段都要求：代码可运行、文档同步、测试/构建可验证

## 已确认决策
1. 总视觉方向：采用“**战术板 + 年鉴**”
2. 一级导航：总览 / 名册 / 排阵 / 战术场景 / 数据中心 / 设置
3. 首页定位：升级为“总控台”，不再只是旧 manager 的直接入口
4. 实施起点：**Phase 1 — 全局壳层重建**（由代理建议并采用）

## 为什么先做全局壳层
先做全局壳层，而不是直接做首页内容，有 4 个好处：
- 先统一第一印象：解锁页、顶部导航、页面容器、按钮/卡片/标签语言会先一致
- 给首页总控台提供稳定骨架：后续做首页摘要、快捷操作、风险卡时不用重复返工
- 最适合当前混合架构：可以先包住 legacy manager，而不用立刻重写内部功能
- 风险最低：改的是结构和视觉承载层，不会马上碰最复杂的业务交互
